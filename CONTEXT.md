# Project Context — GB Train Performance Dashboard

Pick this up in a new Claude Code session by reading: CLAUDE.md, CONTEXT.md, DECISIONS.md.

## What this is
Real-time web dashboard consuming TRUST Train Movement messages (Kafka feed from Rail Data
Marketplace / Confluent Cloud). Shows live delay leaderboard + per-service event timeline.
Built for control room and performance staff — needs to be fast to read, not beautiful.

## Key things that are NOT obvious from the code

### TRUST ID vs headcode
10-character TRUST ID (e.g. "882S65MZ07") is the unique service identifier.
4-character headcode ("2S65") is displayed for readability but IS NOT UNIQUE — recycled
within the same day/TOC. TRUST ID is the join key everywhere in the data model and UI.

### Eviction logic (post Phase 2 update)
- Non-terminated services: keep ALL messages (no per-service message limit).
- Terminated services: apply Rule 1 (latest-N) to free memory.
- Rule 2 (max age): drop any train_id whose most-recent message is older than
  WINDOW_MAX_AGE_MINUTES (default 240 min). Applies regardless of termination status.
- Rationale: users expect to see all station calls for an active service.
  Truncating to 10 messages created confusing incomplete timelines in the drilldown.

### Delay calculation
- Primary: actual_timestamp_ms - gbtt_timestamp_ms
- Fallback when gbtt absent/zero: actual_timestamp_ms - planned_timestamp_ms
- Exclude: offroute_ind == "true" (no valid timetabled reference off-route)
- Only for: 0003 messages with delay_monitoring_point == "true"
- TRUST rounds actual_timestamp to whole minutes → ties possible in qualifying messages;
  use max() not >= loop for consistent tie-breaking in both aggregation and QC runner.

### Known data quirks
- BST bug: TRUST timestamps 1hr ahead during BST. Delay arithmetic is fine (cancels).
  Absolute time display will be 1hr off during BST (accepted v1 limitation).
- Batch Kafka payloads: all items in a list payload share the same received_at timestamp.
- 0007 Change of Identity: service gets new train_id mid-journey → appears as new entry.

### Reference data
- CORPUSExtract.json: {"TIPLOCDATA": [...]} with STANOX + NLCDESC fields. 100% hit rate.
- toc_ref.csv: generated by scripts/fetch_toc_ref.py (scrapes Open Rail Data wiki).
  Community-maintained, may need manual corrections.

## API endpoints
- GET /               Frontend
- GET /api/performance  Leaderboard (params: toc_id, min_delay, limit, include_terminated, include_cancelled)
- GET /api/tocs         All distinct TOC IDs + names currently in store
- GET /api/service/{train_id}  Full event timeline
- GET /api/health       Consumer + reference data status
- GET /api/qc/run       Run 6 QC checks (plain text report)
- GET /api/qc/{train_id}  Full reconciliation trace for one service
